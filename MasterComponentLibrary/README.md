# 🌐 Master Component Library (MCL) Architecture\n\n## 🎯 Purpose\nThis library defines the modular, state-driven architecture for all main video series components. Every element—from visual transitions to CTA activation—must be driven by defined JSON schemas and controlled by a central State Manager.\n\n## 📁 Structure Overview\n*   `src/`: 컴포넌트별 실제 구현 코드 (React/Vue 등 프레임워크 기반의 컴포넌트 스켈레톤)\n*   `schema/`: 모든 컴포넌트의 데이터 구조 정의 (JSON Schema). 이 파일이 '진실'입니다. API 호출 파라미터를 정의합니다.\n*   `assets/`: 재사용 가능한 미디어 에셋 (애니메이션 루프, 사운드 이벤트 등).\n*   `tests/`: 통합 테스트 케이스 및 시나리오 데이터. 컴포넌트가 특정 상태 전환을 정확히 수행하는지 검증합니다.\n*   `api/`: 내부 API 계약 정의 파일 (예: `transition_handler.js`).\n\n## ⚙️ Core Logic Flow (State Machine)\n1.  **Initial State:** `HOOK_STATE` (진입 파라미터: $P_{initial}$) -> 로직 실행 \n2.  **Transition Point:** $P$가 임계값 $\theta_{failure}$ 이하로 하락 시, 상태 변경을 강제합니다.\n3.  **Failure State:** `FAILURE_COMPONENT` (데이터 소스: `schema/failure_data.json`) -> 충격 효과 재생 \n4.  **Recovery Path:** 논리적 해결책 제시 $\rightarrow$ `RECOVERY_COMPONENT` \n5.  **Final CTA:** 지식적 결핍 유도 및 전환 ($LTV_{trigger}$) $\rightarrow$ 최종 이탈 방지 로직.\n\n## 💡 Integration Note\n모든 컴포넌트 호출은 반드시 중앙 State Manager를 통해 이루어져야 하며, 외부에서 임의로 상태를 건너뛰는 것은 시스템 오류(Bug)로 간주합니다. 모든 컴포넌트는 `component_id`와 `required_params`를 명시해야 합니다.\n
