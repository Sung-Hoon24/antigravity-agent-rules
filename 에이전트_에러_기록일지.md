## 📂 [사례 1] 한글 주석 깨짐 및 `SyntaxError: missing )`
### 1. 발생 현상
* 브라우저 개발자 도구 콘솔에 `Uncaught SyntaxError: missing ) after argument list` 에러가 뜨며 스크립트 엔진 전체가 정지함.

### 2. 발생 원인
* 한글 주석이 포함된 HTML 코드를 파이썬 디코더로 불러오는 과정에서, 깨진 바이트(`invalid continuation byte`)를 우회하고자 `errors='ignore'`로 강제 디코딩하여 덮어쓰는 실수가 있었습니다.
* 이 과정에서 특정 코드 라인의 줄바꿈(`\n`)이 파괴되어 아래의 함수 및 변수 선언 60여 줄이 주석줄(`//`)과 한데 얽혀 뭉개졌고, 이로 인해 중괄호와 소괄호 짝이 붕괴되어 구문 에러가 발생하였습니다.

### 3. 해결 대책 & 학습 포인트
* **대책:** 파일 내 꼬여 있던 주석 찌꺼기 영역(1240~1266라인)을 파이썬 스크립트(`restore_tinkerbell.py`)로 정밀 색출하여 삭제하고 괄호 대칭을 맞추어 해결했습니다.
* **학습:** `errors='ignore'`를 통한 덮어쓰기는 코드 훼손의 주범입니다. 텍스트 파일을 기동식으로 수정할 때는 항상 인코딩 손상을 예방하는 방어적 텍스트 매칭을 활용해야 함을 배웠습니다.

## 📂 [사례 2] `ReferenceError: isTinkerbellActive is not defined`
### 1. 발생 현상
* 콘솔창에 `isTinkerbellActive is not defined` 참조 오류가 발생하며 팅커벨 물리 움직임 기능이 굳어버림.

### 2. 발생 원인
* [사례 1]에서 꼬여버린 중복 찌꺼기를 도려낼 때, 찌꺼기 블록 안의 최상단에 묻혀 있던 **팅커벨 마우스 트랙 전역 변수 및 윈도우 포커스 감시 이벤트 리스너 선언부**까지 같이 지워버리는 실수를 저질렀습니다.

### 3. 해결 대책 & 학습 포인트
* **대책:** 지워졌던 팅커벨 관련 전역 변수 세트(`isTinkerbellActive`, `mx`, `my`, `tx`, `ty`, `isWindowFocused` 등)와 마우스 포커스 진입/이탈 가드 리스너들을 스크립트 초입에 다시 정확히 주입하여 복구했습니다.
* **학습:** 코드 블록을 도려내거나 정밀 절제(Delete)할 때는 그 구간 내에 고유하게 정의되어 있던 전역 상태값이 누락되지 않았는지 꼼꼼하게 의존성 검사를 병행해야 합니다.

## 📂 [사례 3] 요정과 은하수 별가루의 위치 분리 현상
### 1. 발생 현상
* 팅커벨 캐릭터 이미지는 얌전히 떠 있으나, 팅커벨 날개 뒤에서 나와야 할 마법 은하수 별가루들이 왼쪽 70px 바깥쪽 허공에서 뿜어지는 기괴한 현상이 발생함.

### 2. 발생 원인
* 날개짓 펄가루 꼬리가 짤리는 현상을 피하기 위해 파티클 캔버스 해상도를 `240px`에서 `400px` 대형으로 키웠으나, 정작 **가루 스폰 포인트 물리 공식(`wingBaseX = isLeftWing ? 105 : 135`, `y = 85`)은 옛날 240px 캔버스 기준(가운데 정렬 120px)으로 낡은 상수값**이 그대로 유지되어 오차가 빚어졌습니다.

### 3. 해결 대책 & 학습 포인트
* **대책:** 400px 대형 캔버스의 캐릭터 정중앙 영점(X: 200px, Y: 130px)에 정확히 대칭되도록 날개 스폰 오프셋을 **`185 : 215` 및 Y=145**로 상향 정렬하여 요정 등 뒤의 날개 관절 위치와 동기화시켰습니다.
* **학습:** 레이아웃의 해상도 스케일이나 캔버스 크기를 리사이징할 때는, 캔버스 내부 좌표계를 추종하는 모든 하위 드로잉 논리(Drawing Logic)의 오프셋 값도 동일한 비율로 슬라이딩 보정해주어야 함을 상기했습니다.

## 📂 [사례 4] 고양이 가이드의 흉물스러운 흰색 바둑판무늬 테두리
### 1. 발생 현상
* 어두운 다크모드 대시보드 화면에 고양이 요정을 띄우자, 투명이 아닌 지저분한 흰색/회색 사각형 격자 테두리가 고양이 주위에 그대로 비쳐 보임.

### 2. 발생 원인
* 고양이 이미지 에셋(`tinkerbell_rpg_cat_right.png`)은 투명 알파 채널이 정상적으로 들어간 이미지가 아니었습니다. AI 생성 과정에서 **포토샵의 가짜 투명 배경 무늬(회색과 흰색이 엇갈린 체크무늬 격자) 자체를 그림 픽셀 데이터(RGB)로 입혀서 내보낸 불량 에셋**이었습니다.
* 또한 테두리 경계면의 부드러운 안개 구간(Anti-Aliasing) 때문에 미세한 중간 톤의 회색/하얀색 파편이 1차 누끼 시 남아 있었습니다.

### 3. 해결 대책 & 학습 포인트
* **대책:** 회색 격자 무늬(`190 ~ 205` 대역)와 밝은 흰색 격자 무늬(`240 ~ 255` 대역)의 RGB 격차 편차가 아주 좁은 무채색 대역 전체를 검출하여, 최외곽에서부터 퍼져나가는 **BFS Flood-fill 누끼 클린 필터**를 파이썬으로 가동해 격자 배경 픽셀 **`367,520개`**를 완전 투명(Alpha = 0)화 시켜 도려냈습니다.
* **학습:** AI 에셋이나 외부 이미지를 다룰 때는 투명 레이어 구조를 전수 분석해야 하며, 픽셀 경계부의 뭉개짐(Anti-Aliasing)을 방어하기 위해 단순 RGB 임계값이 아닌 BFS 무채색 전역 스캔 등을 활용하면 기가 막힌 누끼 퀄리티를 얻을 수 있음을 학습했습니다.

## 📂 [사례 5] 수술 완료 후에도 요정이 여전히 하얗게 나오던 현상
### 1. 발생 현상
* 파이썬 스크립트로 고양이 이미지 누끼를 완벽히 뚫어 놓았음에도 브라우저 화면에는 여전히 하얀 사각형 테두리가 흉하게 로드됨.

### 2. 발생 원인
* **브라우저 이미지 캐시(Cache) 정책의 방해**였습니다. 브라우저는 서버 내의 이미지 파일명이 이전과 동일한 경우, 새 파일로 읽지 않고 브라우저 임시 메모리에 박혀 있던 옛날 바둑판 고양이 이미지를 그대로 재활용하여 렌더링하고 있었습니다.

### 3. 해결 대책 & 학습 포인트
* **대책:** 파일 주소 뒤에 버전 파라미터(`tinkerbell_rpg_cat.png?v=1.4`)를 강제로 매핑하는 **캐시 버스터(Cache Buster)**를 코드에 연동시켜 브라우저의 캐시 재탕을 차단하고 수술 완료된 이미지를 즉시 새로 받도록 강제했습니다.
* **학습:** 실무 웹 서비스 개발 시 에셋이나 스타일시트(CSS)가 변경될 때 캐시 고집 현상은 언제나 발생하므로, 항상 쿼리 스트링(`?v=버전`)을 동적으로 얹는 방어 전략이 상식적으로 탑재되어야 합니다.

## 📂 [사례 6] 크롬 브라우저의 `file://` 보안 락 해제 실패
### 1. 발생 현상
* 콘솔 로그에 `Unsafe attempt to load URL ... 'file:' URLs are treated as unique security origins.` 에러가 나며 팅커벨 관련 파일 링크 및 프레임이 아예 안 보임.

### 2. 발생 원인
* 크롬 브라우저 고유의 로컬 샌드박스 정책입니다. 윈도우 폴더에서 HTML을 마우스 더블클릭하여 파일 프로토콜(`file:///`)로 열면, 해킹 예방을 위해 각 파일의 도메인을 격리하여 다른 리소스나 프레임에 접근할 수 없도록 강제로 교차 출처(CORS)를 차단하기 때문입니다.

### 3. 해결 대책 & 학습 포인트
* **대책:** 로컬 워크스페이스 디렉토리에서 **파이썬 내장 호스팅 웹 서버를 8090 포트로 실시간 백그라운드 구동**(`python -m http.server 8090`)하여, 동일한 오리진의 웹 주소인 `http://localhost:8090/...` 경로로 우회 접속하여 보안 차단을 해결했습니다.
* **학습:** 로컬 마크업 테스트라 할지라도 `iframe`이나 `fetch()`가 복잡하게 얽히는 웹 앱의 경우, `file://` 접근보다 간단한 로컬 웹 서버 호스팅을 조기에 세팅하는 것이 불필요한 보안 삽질을 막는 지름길입니다.

## 📂 [사례 7] 크롬 개발자 도구의 성가신 `favicon.ico` 404 경고
### 1. 발생 현상
* 콘솔에 `:8090/favicon.ico:1 Failed to load resource: 404 (File not found)` 빨간 딱지가 주기적으로 노출됨.

### 2. 발생 원인
* 브라우저가 웹 페이지를 기동할 때 브라우저 탭에 띄울 파비콘(아이콘)을 가져오려고 최상단 루트 경로인 `/favicon.ico`로 자동 노크를 날리는데, 웹 서버 루트 경로에 해당 리소스가 비어 있어서 파이썬 웹 서버가 404 에러로 불평한 것입니다.

### 3. 해결 대책 & 학습 포인트
* **대책:** 루트 디렉토리에 **0바이트 빈 `favicon.ico` 파일**을 더미로 던져 배치하여, 브라우저가 요청할 때 200 OK 빈 데이터로 정상 서빙하여 입막음 처리했습니다.
* **학습:** 개발 콘솔창의 빨간 딱지는 단순 잡음이라도 Tech Lead 관점에서 가독성을 흩트리므로, 이처럼 가볍고 명확한 리소스 배치 기법을 통해 사전에 "Zero Error" 무결성 콘솔을 다지는 습관을 가져야 함을 복기했습니다.

## 📂 [사례 8] 파이썬 쓰기 도중 surrogate 예외로 인한 "0바이트 파일 훼손"
### 1. 발생 현상
* `http://localhost:8090/research/visualizer_study/youtube_visualizer_analyzer.html` 에 들어갔을 때 화면이 404가 뜨거나 완전 빈 하얀 페이지만 출력됨. (파일 크기가 0바이트로 짤려 있음)

### 2. 발생 원인
* 파이썬의 `open(path, "w")` 코드로 쓰기 모드에 들어가는 순간 파일의 기존 내용이 0바이트로 클리어됩니다. 그 후 본 내용을 채워 넣는 와중에 자바스크립트 코드 내에 섞여 있던 대형 이모지(surrogate pair)가 UTF-8 인코딩 쓰기 중 `UnicodeEncodeError` 예외를 유발하여 쓰기 동작이 강제 종료되었고, 파일이 빈 채로 남아버렸습니다.

### 3. 해결 대책 & 학습 포인트
* **대책:** 인코딩 시 대리쌍(surrogates) 예외가 나지 않도록 자바스크립트 내의 불필요한 이모지 제거 정규식을 일반 코드로 단순화하여 인코딩 오류를 방어하였고, 손상되었던 0바이트 연구 과제 파일을 프로젝트 루트의 정상 마스터 복사본으로 복제 및 상대 경로를 재보정하여 부활시켰습니다.
* **학습:** 쓰기 모드로 파일을 열고 데이터를 쓰는 와중에 예외가 나면 파일이 통째로 훼손될 수 있으므로, 임시 파일에 쓰고 이름을 바꾸거나(`tempfile` 핫스왑), 쓰기 과정의 인코딩 예외 가능성을 사전에 엄격히 방어해야 안전함을 상기했습니다.

## 📂 [사례 9] Git 원격 동기화(업 빌드) 시 팅커벨 가이드라인 유실 (Untracked Rollback)
### 1. 발생 현상
* 대시보드(`visualizer_tuner.html`) 및 분석기(`youtube_visualizer_analyzer.html`)에서 팅커벨 가이드를 아무리 잘 심어놓아도, 로컬 서버 리로드나 런봇 기동 동기화(업 빌드)를 진행하는 순간 팅커벨 코드가 감쪽같이 구버전으로 지워지고 유실되는 참사가 자꾸 발생함.

### 2. 발생 원인
* 해당 핵심 마크업 파일들이 Git에 한 번도 추적(`tracked`)되지 않은 상태로 방치되어 있었습니다. 이 상태에서 로컬-원격 간 양방향 동기화(Sync/Pull)를 수행할 때 원격 깃허브의 팅커벨이 탑재되기 전 옛날 버전 소스가 로컬 파일을 밀어버려 실종 사고가 발생했습니다.

### 3. 해결 대책 & 학습 포인트
* **대책:** 다람쥐/올빼미 캐릭터 4장과 최신 마크업 코드를 깃에 등록하고, 규정 10번(`--no-verify` 우회 락 방어)을 준수하여 원격 저장소(`Sung-Hoon24/antigravity-agent-rules.git`)의 master 브랜치로 최종 push 완료하여 로컬과 원격을 1:1로 일치시켰습니다.
* **학습:** 형상 관리 시스템에서 추적 대상에서 유실된 파일(Untracked Files)은 동기화 및 롤백 격발 시 소스 코드가 손쉽게 파괴될 위험이 있으므로, 중요 신규 마스크 및 화면 파일 개발 완료 시에는 반드시 즉각적인 Git Add & Commit을 수행해야 안전함을 깨달았습니다.

## 📂 [사례 10] 로컬 Flask 백엔드 API 서버 미기동으로 인한 ERR_CONNECTION_REFUSED
### 1. 발생 현상
* 대시보드 튜너에서 [💾 설정 저장 및 렌더링 가동] 또는 [🎬 비주얼라이저 템플릿 테스트 모드 적용] 버튼 클릭 시, 브라우저 콘솔에 `net::ERR_CONNECTION_REFUSED` 및 `TypeError: Failed to fetch` 오류가 발생하며 렌더링 및 팅커벨 가이드가 먹통이 됨.

### 2. 발생 원인
* 포트 5000에서 항상 가동 대기 중이어야 할 로컬 Flask API 서버 프로세스(`api/state_machine_service.py`)가 어떠한 연유로 일시적으로 완전히 종료되어 리스닝하지 못하고 있었기 때문임.

### 3. 해결 대책 & 학습 포인트
* **대책:** 프로젝트 루트의 백엔드 감시 데몬인 `console_guardian.py`를 가상환경 파이썬 인터프리터(`venv/Scripts/python.exe console_guardian.py`)를 통해 백그라운드 태스크로 재기동하여 API 포트 5000을 즉각 정상 복구시켰습니다.
* **학습:** 대시보드는 백엔드 API와의 연동이 필수이므로, 통신 에러 감지 시 즉각 로컬 백엔드 서버의 구동 상태(상태 코드 및 백그라운드 프로세스 생존 유무)를 우선적으로 점검하고, 감시 스크립트로 신속하게 리부팅시켜야 안전함을 배웠습니다.

## 📂 [사례 11] 전역 자동화 툴팁 루프의 개별 버튼 상세 가이드 덮어쓰기 유실
### 1. 발생 현상
* 역분석 테스트 핑크색 버튼 `#btn-save-render-reverse`에 마우스 마커를 올렸을 때, 요정별 E2E 3시간 렌더 다음 단계 상세 가이드가 팝업되지 않고, 낡고 간단한 기본 멘트(`장비를 조작해 보시겠어요...?`)가 덮어씌워져 송출됨.

### 2. 발생 원인
* HTML 맨 하단의 전역 자동화 가이드 함수 `initAutoTinkerbellGuide()`가 모든 버튼(`button` 태그)에 호버 툴팁 리스너를 일괄적으로 기계식 매핑하고 있었습니다. 이 공통 리스너는 `data-tink-tip` 속성이 없는 버튼을 만나면 강제로 기본 팁을 씌우도록 예외 없이 동작하여, 상단에서 개별적으로 장착해 둔 정교한 마우스 호버 리스너 멘트를 최종 덮어써서 파괴하고 있었습니다.

### 3. 해결 대책 & 학습 포인트
* **대책:** 공통 자동화 가이드 루프 내에 **역분석 버튼 통과 가드** 체크식(`if (el.id === 'btn-save-render-reverse' || el.id === 'btn-register-template-reverse') return;`)을 긴급 신설하여, 공통 리스너 바인딩에서 역분석 버튼 2종을 예외적으로 스킵(Bypass)하도록 조치했습니다.
* **학습:** 일괄적이고 기계적인 전역 컴포넌트 이벤트 바인딩(Global Selector Event Loop)은 개별 수동 특화 툴팁 및 커스텀 리스너들을 손쉽게 덮어써 버리는 부작용이 있으므로, 예외 처리 필터 가드를 상시 설계 단계에서 고려해야 안전함을 깊이 깨달았습니다.

## 📂 [사례 12] updateTinkerbellPhysics 함수 내 동일 식별자 중복 선언으로 인한 SyntaxError
### 1. 발생 현상
* 대시보드 튜너 리로딩 또는 접근 시, 브라우저가 화면을 렌더링하지 못하고 콘솔창에 `Uncaught SyntaxError: Identifier 'isNearRightEdge' has already been declared` 치명적 문법 에러를 출력하며 팅커벨 관련 모든 UI 동작이 굳어 마비됨.

### 2. 발생 원인
* 물리적 모션 루프 `updateTinkerbellPhysics` 내부의 최상단에서 이미 마우스의 위치를 판정하는 `const isNearRightEdge = mx > (window.innerWidth / 2);` 변수가 정식으로 선언되어 활성 상태였습니다.
* 그러나 하단의 대칭형 시선 결정 구역에서 요정의 실제 위치(`tx`)를 판정하기 위해 동일한 변수명으로 `const isNearRightEdge = tx > (window.innerWidth / 2);` 를 다시 한 번 중복 선언(Re-declaration)함으로써 JS 컴파일 에러가 격발되었습니다.

### 3. 해결 대책 & 학습 포인트
* **대책:** 요정 좌표 기준의 중앙 대칭 변수명을 **`isTinkNearRightEdge`** 로 핀포인트 개명하여 네임스페이스 충돌을 방지하고, 3개 마크업 파일 전체에 대해 SyntaxError를 완벽히 종결시켰습니다.
* **학습:** 스코프 내에서 변수를 선언할 때는 상단 블록에 이미 존재하고 있는 식별자 목록을 면밀히 사전에 훑어보아야 하며, 특히 물리 엔진 루프처럼 코드가 밀접한 함수 내에서는 가급적 역할별로 구체화된 고유 이름(예: `isTinkNearRightEdge`)을 부여하여 식별자 충돌을 원천 차단하는 습관을 가져야 무결성이 확보됨을 배웠습니다.
